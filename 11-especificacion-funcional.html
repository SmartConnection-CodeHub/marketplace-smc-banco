01Premisa

Doc 11 detalla qué hace el sistema desde la mirada operativa de los 3 founders. Cada caso de uso es una acción real que Camila · Javier o Guillermo ejecutan en el sistema. Las reglas de negocio son las invariantes que el sistema debe garantizar siempre.

💡 División funcional vs técnico

Este doc es funcional puro. Lo técnico (catálogo de APIs, maquetas wireframe, SQL queries de aceptación) vive en Doc 13 Especificación Técnica. Si necesitás endpoints, schemas o specs de UI, andá al 13.

Cada caso de uso (CU) tiene 3 atributos visibles: Trigger (qué lo dispara), Output (qué deja como resultado), Tools (qué herramientas del stack participan).

0518 reglas de negocio · matriz visual

Reglas invariantes del sistema. Cada BR es un contrato que el sistema NO viola nunca. Rojo = crítica (puede romper compliance o reputación).

BR-001 · CRÍTICA
Fotos pro propias siempre
Validation requiere >=3 fotos · detector perceptual hash bloquea fotos genéricas de proveedor
BR-002
Stock real-time cross-canal
Inventory master · adapters sync vía pg_notify · timeout 1s · alerta CloudWatch si excede
BR-003
Channel Adapter pattern
Cada canal (MeLi · D2C · MP · B2B) implementa interfaz uniforme · plug-and-play
BR-004
Supplier Adapter pattern
Cada proveedor (Nexport · local · dropship) implementa interfaz uniforme
BR-005 · CRÍTICA
pg_notify cross-canal stock <1s
Trigger postgres ON UPDATE inventory · NOTIFY channel-sync · listener Node.js suscrito
BR-006 · CRÍTICA
SELECT FOR UPDATE stock
Reserva atómica al recibir orden · evita oversell concurrente · rollback si falla pago
BR-007
Webhooks firmados HMAC SHA-256
MeLi · MP · pagos · couriers · todos validan firma antes de procesar
BR-008
Idempotency key obligatoria
Header X-Idempotency-Key en webhooks · evita doble cobro · doble DTE
BR-009 · CRÍTICA
DTE automático post-venta
Evento order.completed dispara emit DTE · <30s end-to-end · retry 5x si rechazo
BR-010
DTE 39 default · 33 si RUT empresa
Si customer_rut termina en -K o letra → DTE 33 · sino DTE 39 · 60-66M forzado 33
BR-011
Retry exponencial SII
5 intentos · 1s · 2s · 4s · 8s · 16s · si falla → cola manual + alerta
BR-012
RLS Supabase tenant_id
Toda tabla tiene tenant_id · RLS policy filtra por sesión · multi-tenant ready Y3+
BR-013
Cerebro rate limit 60/día/founder
Counter Postgres incrementa por mensaje · reset 00:00 CL · evita gasto LLM
BR-014
Refresh MeLi tokens 6h
Cron Vercel cada 6h · refresh OAuth · si falla 3x → alerta a Guillermo
BR-015
Sync MP 8am AM diario
Cron 8am · ticket auth · si MP devuelve >100 licitaciones → filtrar via Cerebro top 20
BR-016 · CRÍTICA
Fallback chain IA · NUNCA Claude
OpenRouter: Groq → Gemini → DeepSeek · Claude excluido explícito (regla Hoku)
BR-017
Cache hot 6h · warm 12h · cold 7d
Sistema Oportunidades · max 1000 filas total · pg_cron cleanup diario
BR-018
Auditoría append-only
Tabla webhook_events · order_events · invoice_events · solo INSERT · nunca UPDATE/DELETE

06Out of scope explícito Y1
