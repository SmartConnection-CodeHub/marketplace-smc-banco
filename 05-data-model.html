🎯 Reglas obligatorias toda tabla
  • id UUID PRIMARY KEY DEFAULT gen_random_uuid()
  • tenant_id UUID NOT NULL REFERENCES mk_tenants(id) (excepto mk_tenants)
  • created_at TIMESTAMPTZ DEFAULT NOW()
  • updated_at TIMESTAMPTZ DEFAULT NOW()
  • created_by UUID REFERENCES mk_users(id) (auditoría)
  • Index sobre (tenant_id, created_at) obligatorio
  • RLS policy: USING (tenant_id = current_setting('app.current_tenant')::uuid)

02Multi-tenancy · pieza central

Cada tenant = una "empresa operadora" dentro del Hub. Hoy hay 1 tenant (Smart Connection / Marketplace SMC SpA). Futuro: Kanki + N marcas adicionales.

Beneficios concretos:

  • 1 sola BD · 1 sola app desplegada
  • Aislamiento garantizado via RLS Postgres (tenant A NUNCA ve data tenant B)
  • Agregar tenant nuevo = 1 INSERT en mk_tenants · no es deploy nuevo
  • Métricas consolidadas posibles (admin del Hub puede ver todos)
Marketplace SMC · Blueprint v1.004

03Tablas Core

mk_tenants
Una fila = una empresa operadora dentro del Hub.
id UUID PK
slug TEXT UNIQUE              -- 'smc' · 'kanki' · 'marca-x'
name TEXT                     -- razón social legal
rut TEXT                      -- RUT empresa
giro TEXT                     -- giro principal SII
domicilio JSONB               -- dirección estructurada
plan TEXT                     -- 'mvp' · 'standard' · 'enterprise'
settings JSONB                -- configuración tenant-específica
active BOOLEAN DEFAULT true
created_at TIMESTAMPTZ
updated_at TIMESTAMPTZ
mk_users
Operadores del Hub. Pueden tener acceso a múltiples tenants.
id UUID PK
email TEXT UNIQUE
full_name TEXT
auth_id UUID                  -- ref Supabase Auth
role TEXT                     -- 'admin' · 'operator' · 'viewer' · 'accountant'
active BOOLEAN DEFAULT true
created_at TIMESTAMPTZ
mk_user_tenants
Many-to-many · qué usuarios acceden a qué tenants con qué rol.
user_id UUID FK → mk_users
tenant_id UUID FK → mk_tenants
role TEXT                     -- rol específico en este tenant
PRIMARY KEY (user_id, tenant_id)
mk_categories
Categorías de productos · jerarquía padre-hijo · compartida cross-tenant si flag.
id UUID PK
tenant_id UUID           -- NULL si es categoría global SMC
slug TEXT
name TEXT
parent_id UUID                -- self-ref para jerarquía
meta JSONB

04Tablas Inventory · el corazón

mk_products
SKU master. UN producto = UNA fila aquí. Se publica en N canales.

sku TEXT NOT NULL             -- código interno
ean_upc TEXT                  -- código barras

status TEXT                   -- 'active' · 'discontinued' · 'draft'
UNIQUE (tenant_id, sku)

Bodegas físicas del tenant. Hoy 1 · Y2 múltiples.

tenant_id UUID
name TEXT                     -- 'Casa Matriz' · 'Bodega Sur'

Stock físico por SKU por bodega. SSoT del inventario.

mk_inventory_movements
Log de TODO movimiento de stock. Auditable. Event-sourced.

notes TEXT

09Tablas Eventos & Audit

mk_evt_operations (event store)
Log de TODOS los eventos operativos. Sirve como event-source para reconstruir estado · alimenta IA · audita procesos.
id UUID PK
tenant_id UUID
event_type TEXT               -- 'order.created' · 'inventory.received' · etc
aggregate_type TEXT           -- 'order' · 'po' · 'inventory'

created_at TIMESTAMPTZ

tenant_id UUID
table_name TEXT
row_id UUID

changed_by UUID

created_at TIMESTAMPTZ

Log llamadas API externas e internas. Reusa concepto actual de app marketplace.
