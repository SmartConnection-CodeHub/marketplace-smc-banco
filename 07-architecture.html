04Event-driven · cómo funciona

Cada operación significativa emite un evento. Los módulos suscritos reaccionan automático. Esto evita acoplamiento (módulo finance no llama a inventory · escucha eventos).

💡 Implementación pragmática

No usar Kafka/Redis al inicio · es overkill. Usar Postgres NOTIFY + tabla mk_evt_operations como event store + pg_cron para retries. Cuando volumen pase 10K eventos/día evaluar Inngest o Trigger.dev.

05Multi-tenancy técnico · Row Level Security

Setup RLS Postgres

-- En cada query, el server setea el tenant context
SET app.current_tenant = '<tenant-uuid>';

-- Policy aplicada a TODAS las tablas (excepto mk_tenants y mk_users)
CREATE POLICY tenant_isolation ON mk_products
  USING (tenant_id = current_setting('app.current_tenant')::uuid)
  WITH CHECK (tenant_id = current_setting('app.current_tenant')::uuid);

ALTER TABLE mk_products ENABLE ROW LEVEL SECURITY;
ALTER TABLE mk_products FORCE ROW LEVEL SECURITY;

Flujo request

  1. Operador hace login · recibe JWT que incluye tenant_id
  2. Cada request incluye header X-Tenant-Id
  3. Middleware Next.js valida JWT + tenant_id match con permisos del usuario
  4. Middleware setea SET app.current_tenant = ? en conexión Postgres
  5. Cualquier query subsiguiente está auto-filtrada por RLS
  6. Imposible (técnicamente) que tenant A vea data de tenant B aunque haya bug en código aplicación
✅ Beneficio defensa en profundidad

Aunque un dev olvide filtrar WHERE tenant_id = ? · RLS Postgres bloquea. Defensa de último nivel.

Pipeline CI/CD

push branch → GitHub Actions:
   1. install (cache pnpm)
   2. lint (eslint)
   3. type-check (tsc --noEmit)
   4. unit tests (vitest run)
   5. build (next build)
   6. integration tests (vitest --integration)
   7. e2e smoke (playwright · solo en main)
   8. deploy (auto vía Amplify webhook)
   9. health check post-deploy (curl /api/health)
   10. rollback automático si health falla

Migraciones BD

  • SQL versionado en supabase/migrations/YYYY-MM-DD-name.sql
  • CI corre migraciones contra staging primero · verifica · luego PRD
  • Rollback scripts obligatorios para cada migration
  • Zero downtime · todas las migraciones backward-compatible (additive)

08Seguridad operativa

  • Secrets: AWS Secrets Manager (Amplify env vars en plano solo en MVP · migrar a Secrets Manager Q2)
  • OAuth tokens externos: cifrados at rest con pgcrypto en mk_channels.credentials
  • Refresh tokens: cron auto-renueva antes de expirar (existe lib en código actual)
  • HTTPS: obligatorio · HSTS · TLS 1.3 mínimo · cert auto AWS ACM
  • CSP: Content Security Policy estricta · no inline scripts
  • Rate limiting: por IP en endpoints públicos · por tenant en API interna
  • Audit trail: mk_audit_changes en cada UPDATE/DELETE de tablas financieras
  • 2FA: opcional MVP · obligatorio para rol admin desde Fase 2
  • Backups BD: Supabase point-in-time recovery 7 días (plan Pro) · snapshot diario S3
  • Penetration testing: antes de Fase 4 (SaaS público) · vendor externo
  • SOC2 readiness: documentar controles desde día 1 · si se vuelve SaaS · auditoría Q4 año 2
